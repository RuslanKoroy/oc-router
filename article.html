oc-router: диспетчер моделей для OpenCode

Проект oc-router решает простую, но дорогую проблему: не каждую просьбу к AI‑ассистенту нужно отдавать самой мощной модели. Иногда нужен быстрый разведчик, иногда аккуратный универсал, иногда тяжёлая артиллерия.

ЯзыкTypeScript / ESM
Интеграция@opencode-ai/plugin
Рольумная маршрутизация запросов

Эта статья объясняет проект от человеческой причины до конкретных файлов: src/plugin.ts, src/router.ts, src/config.ts, src/install.ts и остальных модулей.

Часть 1

Зачем это существует

Представьте команду разработчиков, которая постоянно пользуется AI‑помощником в редакторе. Они просят его посмотреть файлы, объяснить ошибку, написать тест, спроектировать новую архитектуру, провести ревью безопасности. Все эти просьбы звучат одинаково: человек пишет текст. Но внутри они очень разные по цене ошибки.

До oc-router пользователь обычно выбирал модель вручную. Это похоже на гараж, где есть велосипед, обычная машина и грузовик, но каждая поездка начинается с вопроса: «На чём ехать?» За хлебом на грузовике — дорого и медленно. Перевозить пианино на велосипеде — опасно. А если выбирать некогда, люди оставляют один вариант по умолчанию и переплачивают либо рискуют качеством.

oc-router ставит между человеком и моделями диспетчера. Пользователь пишет обычную просьбу, а диспетчер решает: это работа для быстрого дешёвого помощника, для основной модели или для самой сильной модели.

Главная идея проекта: модель выбирается не по привычке пользователя, а по смыслу задачи.

Один конкретный сценарий

Разработчица Лена работает в большом репозитории. Утром она пишет: «Найди, где формируется конфиг роутера». Это безопасная разведка: достаточно прочитать файлы и составить карту. oc-router отправляет запрос в tier fast.

Через десять минут Лена пишет: «Исправь баг в fallback‑логике и добавь тест». Это уже изменение кода: нужен tier balanced. После обеда она просит: «Оцени архитектуру делегирования между агентами и предложи безопасный редизайн». Это дорогая задача с риском неверного решения — запрос уходит в large.

Для Лены интерфейс остаётся тем же. Меняется только внутренний исполнитель.

Часть 2

Как это выглядит снаружи

Снаружи проект выглядит как плагин OpenCode и отдельная команда oc-router. Плагин работает во время чата, а CLI помогает установить, настроить и диагностировать систему.

Что делает пользователь

  • Запускает oc-router init, чтобы пройти интерактивную настройку.
  • Смотрит доступные модели через oc-router models.
  • Проверяет состояние через oc-router status или oc-router doctor.
  • Меняет параметры через oc-router config get и oc-router config set.
  • В OpenCode пишет обычные сообщения, иногда принудительно добавляя /fast, /balanced или /large.
Почему так спроектировано: пользователь не должен изучать внутренности маршрутизатора ради обычной работы. Но когда нужно вмешаться, есть явные команды, конфиги и принудительные префиксы.

Поверхность конфигурации

У проекта две точки хранения настроек: глобальная ~/.config/oc-router/config.json и проектная .opencode/router.json. Проектная настройка сильнее глобальной: это позволяет иметь общую привычную схему и переопределять модели в конкретном репозитории.

.opencode/router.jsonпример идеи конфигурации
Конфиг описывает не только модели, но и поведение дисплея, маршрутизации и fallback.

2  "enabled": true,

Часть 3

Большая картина изнутри

Внутри oc-router состоит из четырёх больших областей: плагина OpenCode, маршрутизатора решений, конфигурационной системы и CLI/инсталлятора. Пятый слой — агенты tier‑ов — создаётся плагином как часть OpenCode‑окружения.

Главный вход — src/index.ts, который экспортирует плагин. Реальная работа сосредоточена в src/plugin.ts: там OpenCode получает hooks, агенты, системные подсказки и правила отображения.

config hookchat.message hookexperimental.text.completetool.execute.afterexperimental.chat.system.transform

src/router.ts принимает пользовательский текст и конфиг, затем возвращает RoutingResult: какой tier выбран, какую модель поставить, почему, с какой уверенностью и был ли это принудительный выбор.

Часть 4

Ключевые механизмы через аналогии

1. Router model classification

Аналогия. Это диспетчер в приёмной больницы. Он не лечит пациента, а решает, куда направить: быстрое окно справок, обычный кабинет или реанимационная команда.

Зачем существует. Без него пользователь сам выбирает модель для каждой задачи. Это утомительно и приводит к двум ошибкам: дорогая модель делает простую работу или слабая модель берётся за рискованную задачу.

Как работает технически. routeRequest() в src/router.ts сначала проверяет принудительные префиксы. Если их нет, строится prompt через buildRouterPrompt() из src/prompts.ts. Router model должна вернуть JSON по схеме routerDecisionSchema: tier, reason, confidence. Затем tier сверяется с конфигом, уверенность сравнивается с routing.minConfidence, и при сомнениях включается fallback.

2. Delegation protocol

Аналогия. Это инструкция начальнику смены: «Ты не должен сам чинить всё подряд. Осмотри заявку, выбери правильную бригаду и передай ей задачу».

Зачем существует. OpenCode имеет агентную систему. Проект использует её не как украшение, а как дисциплину: @router — оркестратор, @fast, @balanced, @large — исполнители разных классов задач.

Как работает технически. В src/plugin.ts функция buildDelegationProtocol(config) генерирует обязательный текст системного протокола. Он вставляется через experimental.chat.system.transform. В config hook регистрируются агенты с описаниями, моделями и правами.

src/plugin.tsbuildDelegationProtocol(config)
Протокол описывает роли и требует handoff между агентами.

2  const tiers = configuredTiers(config)
3    .map(([name, tier]) => `@${name} -> ${tier.model}`)
4    .join("\n")
5  return ["## Model Delegation Protocol — MANDATORY", tiers].join("\n")

3. Context handoff system

Аналогия. Разведчик не просто говорит инженеру «там мост». Он приносит карту, фотографии, список рисков и пометки, какие дороги уже проверены. Инженеру не нужно начинать разведку заново.

Зачем существует. Дорогие модели не должны тратить контекст и деньги на повторное чтение файлов, если дешёвый агент уже собрал факты. Это особенно важно для больших репозиториев.

Как работает технически. Протокол в src/plugin.ts требует, чтобы @fast выдавал блок CONTEXT HANDOFF: какие файлы прочитаны, что найдено, какие риски остались. Инструкции для @balanced и @large говорят использовать этот блок как первичный контекст и не перечитывать файлы без причины.

4. Permission system

Аналогия. Стажёру‑разведчику выдают ключ от архива и фонарик, но не выдают ключ от серверной и сварочный аппарат. Он может смотреть, но не ломать.

Зачем существует. Tier fast предназначен для дешёвой безопасной разведки. Если он случайно начнёт редактировать файлы, экономия превратится в риск.

Как работает технически. В src/plugin.ts для @fast разрешены операции чтения и поиска: read, grep, glob, list, bash. Запрещены edit, write, apply_patch, task. Для @balanced и @large разрешены широкие действия, но task всё равно запрещён, чтобы избежать бесконтрольной рекурсии делегирования.

5. Configuration hierarchy

Аналогия. В компании есть общие правила, но у каждого проекта может быть своя инструкция на двери: «В этом цехе используем другие станки». Проектная табличка уточняет, но не уничтожает корпоративный регламент.

Зачем существует. Один пользователь может работать в разных репозиториях с разными требованиями, бюджетом и доступными моделями.

Как работает технически. src/config.ts хранит defaultConfig, читает глобальный и проектный JSON, затем объединяет их через mergeConfig(). Вложенные секции router, tiers, display, routing сливаются рекурсивно. validateRouterConfig() проверяет формат моделей provider/model.

6. Error resilience with fallback

Аналогия. Если диспетчерская рация сломалась, скорая не останавливается на обочине. Она едет по безопасному маршруту по умолчанию.

Зачем существует. Плагин маршрутизации не должен ломать OpenCode. Если router model недоступна, вернула мусор или confidence слишком низкий, пользователь всё равно должен получить работующую сессию.

Как работает технически. В src/router.ts операции маршрутизации обёрнуты в обработку ошибок. При исключении, невалидном tier или низкой уверенности выбирается fallbackTier, по умолчанию balanced. В баннере можно показать причину fallback.

Часть 6

Логика шаг за шагом

Сценарий 1: первичная установка

  1. Пользователь запускает CLI. Команда входит в src/cli.ts, где выбирается handler для init.
  2. Мастер ищет модели. src/onboarding.ts использует модельный отчёт из src/models-report.ts, который запускает opencode models --verbose.
  3. Пользователь выбирает router и tiers. Для каждого tier выбирается provider/model; при необходимости пользователь вводит строку вручную.
  4. Конфиг записывается. writeRouterConfig() из src/config.ts сохраняет JSON в глобальный или проектный путь.
  5. Инсталлятор создаёт OpenCode‑файлы. src/install.ts пишет plugin loader, agent files и slash commands.

Сценарий 2: автоматическая маршрутизация запроса

  1. Сообщение попадает в OpenCode. chat.message hook из src/plugin.ts получает пользовательский текст.
  2. Загружается конфигурация. loadRouterConfig() объединяет defaults, global config и project config.
  3. Вызов routeRequest(). src/router.ts проверяет режим, enabled flag и forced prefixes.
  4. Router model классифицирует запрос. Prompt строится в src/prompts.ts, ответ проверяется по ожидаемой структуре.
  5. Выбирается tier model. Например, large превращается в конкретный provider/model из config.tiers.large.model.
  6. Плагин меняет модель сообщения. OpenCode продолжает обычное выполнение, но уже с выбранной моделью.
  7. Пользователь видит объяснение. src/display.ts формирует routing banner и, если включено, toast.

Сценарий 3: принудительный override через prefix

  1. Пользователь пишет /large спроектируй стратегию миграции.
  2. routeRequest() смотрит config.routing.forceTierPrefixes, где по умолчанию есть /fast, /balanced, /large.
  3. Prefix удаляется из текста, чтобы целевая модель не видела служебную команду как часть задачи.
  4. Router model не вызывается: решение уже принято пользователем.
  5. RoutingResult помечается как forced, а banner может показать, что выбор был ручным.

Сценарий 4: изменение конфигурации и reload

  1. Пользователь вызывает oc-router config set ... в src/cli.ts.
  2. CLI читает текущий конфиг через loadRouterConfig() или целевой файл напрямую, меняет значение и пишет JSON через writeRouterConfig().
  3. При следующем hook‑вызове plugin снова загружает конфиг. В проекте нет отдельной базы или long‑running config cache, поэтому модель поведения проста: файл — источник правды.
  4. validateRouterConfig() ловит ошибки вроде неверного provider/model.

Сценарий 5: восстановление после ошибки

  1. Router model недоступна, возвращает не JSON или выбирает неизвестный tier.
  2. src/router.ts не выбрасывает ошибку наружу как фатальную для OpenCode‑сессии.
  3. Выбирается config.fallbackTier, по умолчанию balanced.
  4. Пользователь получает рабочий ответ, а не сломанную сессию.
  5. Диагностику можно выполнить через oc-router doctor.

Часть 7

Решения, которые не видны снаружи

Почему три tier, а не два или четыре?

Три — это понятная шкала риска: разведка, обычная работа, сложная работа. Два tier заставили бы смешивать «обычное» и «сложное»; четыре усложнили бы выбор и конфиг без очевидной выгоды.

Почему отдельный CLI?

OpenCode не документирует расширение top‑level CLI плагинами. Поэтому oc-router честно живёт рядом: устанавливает файлы OpenCode и показывает, что именно изменил.

Почему synthetic message parts?

Баннер становится частью видимого разговора и объясняет выбор модели там, где пользователь уже смотрит. Toast хорош для мгновенного сигнала, но баннер остаётся в контексте.

Почему context handoff?

Полное автоматическое разделение контекста между моделями дорого и непрозрачно. Handoff — явный контракт: что прочитано, что найдено, что остаётся неизвестным.

Почему права для @fast?

@fast оптимизирован как дешёвый разведчик. Ограничения превращают это намерение в механизм безопасности, а не просто рекомендацию в prompt.

Почему ошибки не фатальны?

Маршрутизатор — вспомогательный слой. Если он падает, базовый OpenCode должен продолжать работать. Поэтому fallback важнее строгой остановки.

Честный пробел: из переданного контекста не анализировались производительность router model calls, конкурентные запросы, взаимодействие с другими плагинами OpenCode и реальная совместимость с разными версиями OpenCode. Архитектура выглядит рассчитанной на устойчивость, но эти аспекты требуют отдельной проверки.

Часть 8

Как запустить и попробовать

Ниже — практический путь для первого знакомства. Команды основаны на структуре проекта и README‑описании из контекста.

  1. Соберите проект. В корне /home/user128/oc-router выполните npm install, затем npm run build. Ожидаемый результат: TypeScript компилируется в dist/.
  2. Запустите тесты. Выполните npm test. Тесты Vitest покрывают config, router, display, plugin hooks, install и CLI‑поведение.
  3. Инициализируйте настройку. Запустите oc-router init или локальный бинарь после сборки. Мастер должен показать доступные модели OpenCode или позволить ввести ID вручную.
  4. Проверьте конфиг. Выполните oc-router status или oc-router doctor. Вы должны увидеть, включён ли routing, какие модели назначены tier‑ам, и есть ли проблемы.
  5. Попробуйте обычный запрос. В OpenCode напишите просьбу вроде «найди файлы, где описана конфигурация». Должен появиться routing banner с выбранным tier и причиной.
  6. Попробуйте override. Напишите /large объясни архитектурные риски этого проекта. Router model должна быть обойдена, а запрос должен пойти в large.
